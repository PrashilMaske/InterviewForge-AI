# Database Schema Specification - InterviewForge AI

This document specifies the PostgreSQL relational database schema for InterviewForge AI. All fields, relations, keys, indexes, and performance optimizations are modeled below.

---

## 1. Relational ER Schema

The database model is designed to support structured tracking of user profiles, multiple resume files with analytical audits, job descriptions, adaptive mock sessions, and weekly learning paths.

```text
+-------------------+       +--------------------+       +--------------------+
|       users       |-------|<      resumes     |-------|<  resume_analyses  |
+-------------------+       +--------------------+       +--------------------+
          |                           |
          |                           |
          |   +-----------------------+
          |   |
          v   v
+-------------------+       +--------------------+
| interview_sessions|-------|  job_descriptions  |
+-------------------+       +--------------------+
          |
          v
+-------------------+
|     questions     |
+-------------------+
          |
          v
+-------------------+
|      answers      |
+-------------------+
          |
          v
+-------------------+       +--------------------+
|    evaluations    |       |   progress_history |
+-------------------+       +--------------------+
          |                           |
          v                           v
+-------------------+       +--------------------+
|      reports      |       |   learning_plans   |
+-------------------+       +--------------------+
```

---

## 2. Table Specifications

### A. Table: `users`
Tracks user attributes, roles, and interview streaks.
* **Fields**:
  * `id`: `UUID` (Primary Key, defaults to `gen_random_uuid()`)
  * `email`: `VARCHAR(255)` (Unique, Indexed)
  * `username`: `VARCHAR(150)` (Unique)
  * `password`: `VARCHAR(255)`
  * `role`: `VARCHAR(20)` (Constraint: `candidate`, `admin`, `recruiter`)
  * `streak`: `INT` (Default: `0`)
  * `last_active`: `TIMESTAMP WITH TIME ZONE`
  * `is_active`: `BOOLEAN` (Default: `true`)
  * `is_staff`: `BOOLEAN` (Default: `false`)
  * `date_joined`: `TIMESTAMP WITH TIME ZONE` (Default: `CURRENT_TIMESTAMP`)
* **Indexes**:
  * `idx_users_email` (B-Tree on `email`)
  * `idx_users_role` (B-Tree on `role`)

### B. Table: `resumes`
Tracks candidate resume uploads.
* **Fields**:
  * `id`: `UUID` (Primary Key)
  * `user_id`: `UUID` (Foreign Key -> `users.id` ON DELETE CASCADE)
  * `file_url`: `TEXT` (Cloudinary file location)
  * `file_name`: `VARCHAR(255)`
  * `version`: `INT` (Incremental per candidate)
  * `created_at`: `TIMESTAMP WITH TIME ZONE`
* **Indexes**:
  * `idx_resumes_user_id` (B-Tree on `user_id`)
  * `idx_resumes_user_version` (Unique Index on `(user_id, version)`)

### C. Table: `resume_analyses`
Stores the detailed grading output generated by the AI Resume Analyzer.
* **Fields**:
  * `id`: `UUID` (Primary Key)
  * `resume_id`: `UUID` (Foreign Key -> `resumes.id` ON DELETE CASCADE)
  * `overall_score`: `INT` (Range: 0-100)
  * `formatting_score`: `INT`
  * `projects_score`: `INT`
  * `skills_score`: `INT`
  * `ats_score`: `INT`
  * `grammar_score`: `INT`
  * `achievements_score`: `INT`
  * `keywords_score`: `INT`
  * `section_explanations`: `JSONB` (Stores AI explanation fields per score category)
  * `improved_bullets`: `JSONB` (Stores before/after mapped suggestions)
  * `parsed_entities`: `JSONB` (Extracted skills, tools, and experience metrics)
  * `created_at`: `TIMESTAMP WITH TIME ZONE`
* **Indexes**:
  * `idx_resume_analyses_resume_id` (B-Tree on `resume_id`)

### D. Table: `job_descriptions`
Allows matching of resumes against job posts.
* **Fields**:
  * `id`: `UUID` (Primary Key)
  * `user_id`: `UUID` (Foreign Key -> `users.id` ON DELETE CASCADE)
  * `title`: `VARCHAR(255)`
  * `raw_text`: `TEXT`
  * `file_url`: `TEXT` (Optional, Cloudinary URL)
  * `created_at`: `TIMESTAMP WITH TIME ZONE`
* **Indexes**:
  * `idx_job_descriptions_user_id` (B-Tree on `user_id`)

### E. Table: `interview_sessions`
Represents an adaptive mock interview attempt.
* **Fields**:
  * `id`: `UUID` (Primary Key)
  * `user_id`: `UUID` (Foreign Key -> `users.id` ON DELETE CASCADE)
  * `resume_id`: `UUID` (Foreign Key -> `resumes.id` ON DELETE SET NULL)
  * `jd_id`: `UUID` (Foreign Key -> `job_descriptions.id` ON DELETE SET NULL, Nullable)
  * `interview_type`: `VARCHAR(50)` (e.g. `Python`, `System Design`, `Behavioral`)
  * `difficulty`: `VARCHAR(20)` (Constraint: `easy`, `medium`, `hard`, `adaptive`)
  * `current_difficulty_level`: `INT` (Scale 1 to 10, adjusts adaptively)
  * `status`: `VARCHAR(20)` (Constraint: `pending`, `active`, `completed`)
  * `created_at`: `TIMESTAMP WITH TIME ZONE`
  * `ended_at`: `TIMESTAMP WITH TIME ZONE` (Nullable)
* **Indexes**:
  * `idx_sessions_user_id` (B-Tree on `user_id`)
  * `idx_sessions_status` (B-Tree on `status`)

### F. Table: `questions`
Contains questions generated during an interview.
* **Fields**:
  * `id`: `UUID` (Primary Key)
  * `session_id`: `UUID` (Foreign Key -> `interview_sessions.id` ON DELETE CASCADE)
  * `question_text`: `TEXT`
  * `expected_concepts`: `JSONB` (List of target keywords/concepts)
  * `difficulty_level`: `INT` (Stored scaling level at generation time)
  * `created_at`: `TIMESTAMP WITH TIME ZONE`
* **Indexes**:
  * `idx_questions_session_id` (B-Tree on `session_id`)

### G. Table: `answers`
Stores the candidate response to a question.
* **Fields**:
  * `id`: `UUID` (Primary Key)
  * `question_id`: `UUID` (Unique, Foreign Key -> `questions.id` ON DELETE CASCADE)
  * `answer_text`: `TEXT`
  * `response_time_seconds`: `INT`
  * `created_at`: `TIMESTAMP WITH TIME ZONE`

### H. Table: `evaluations`
Stores grading data generated by the AI Answer Evaluator.
* **Fields**:
  * `id`: `UUID` (Primary Key)
  * `answer_id`: `UUID` (Unique, Foreign Key -> `answers.id` ON DELETE CASCADE)
  * `overall_score`: `NUMERIC(4,2)` (Scale 1-10)
  * `technical_score`: `INT`
  * `communication_score`: `INT`
  * `confidence_score`: `INT`
  * `depth_score`: `INT`
  * `strengths`: `JSONB` (List of strings)
  * `weaknesses`: `JSONB` (List of strings)
  * `missing_concepts`: `JSONB` (List of strings)
  * `ideal_answer`: `TEXT`
  * `suggestions`: `JSONB` (List of strings)
  * `created_at`: `TIMESTAMP WITH TIME ZONE`
* **Indexes**:
  * `idx_evaluations_answer_id` (B-Tree on `answer_id`)

### I. Table: `reports`
Summary PDF data generated at the end of the session.
* **Fields**:
  * `id`: `UUID` (Primary Key)
  * `session_id`: `UUID` (Unique, Foreign Key -> `interview_sessions.id` ON DELETE CASCADE)
  * `pdf_url`: `TEXT` (Cloudinary URL)
  * `readiness_score`: `INT` (0-100 scale estimate)
  * `timeline`: `JSONB` (Progress timeline logs)
  * `projects_recommended`: `JSONB` (Recommendations list)
  * `courses_recommended`: `JSONB` (Recommendations list)
  * `created_at`: `TIMESTAMP WITH TIME ZONE`

### J. Table: `learning_plans`
Stores custom roadmaps curated to patch candidate knowledge deficiencies.
* **Fields**:
  * `id`: `UUID` (Primary Key)
  * `user_id`: `UUID` (Foreign Key -> `users.id` ON DELETE CASCADE)
  * `session_id`: `UUID` (Foreign Key -> `interview_sessions.id` ON DELETE CASCADE)
  * `roadmap_data`: `JSONB` (Custom weekly modules structure)
  * `current_week`: `INT` (Default: 1)
  * `total_weeks`: `INT`
  * `created_at`: `TIMESTAMP WITH TIME ZONE`
* **Indexes**:
  * `idx_learning_plans_user_id` (B-Tree on `user_id`)

### K. Table: `progress_history`
Maintains daily historical aggregates for user performance charts.
* **Fields**:
  * `id`: `UUID` (Primary Key)
  * `user_id`: `UUID` (Foreign Key -> `users.id` ON DELETE CASCADE)
  * `date`: `DATE`
  * `resume_score`: `INT` (Nullable)
  * `average_interview_score`: `NUMERIC(4,2)`
  * `skill_strengths`: `JSONB`
  * `skill_weaknesses`: `JSONB`
  * `created_at`: `TIMESTAMP WITH TIME ZONE`
* **Indexes**:
  * `idx_progress_history_user_date` (Unique index on `(user_id, date)`)

## 3. SQL Constraints and Triggers

1. **Self-Incrementing Resume Versioning**:
   A PostgreSQL database trigger dynamically increments `version` for new resumes uploaded by the same user, avoiding race conditions on the Django application side.
   ```sql
   CREATE OR REPLACE FUNCTION set_resume_version()
   RETURNS TRIGGER AS $$
   DECLARE
       next_version INT;
   BEGIN
       SELECT COALESCE(MAX(version), 0) + 1 INTO next_version
       FROM resumes
       WHERE user_id = NEW.user_id;
       
       NEW.version := next_version;
       RETURN NEW;
   END;
   $$ LANGUAGE plpgsql;

   CREATE TRIGGER trg_resume_version_inc
   BEFORE INSERT ON resumes
   FOR EACH ROW
   EXECUTE FUNCTION set_resume_version();
   ```

2. **Cascade Delete Policies**:
   * Deleting a `User` cascades to delete their `resumes`, `job_descriptions`, `interview_sessions`, and `progress_history`.
   * Deleting an `interview_session` cascade deletes all `questions`, `answers`, `evaluations`, and `reports`.
   * Deleting a `resume` sets the `resume_id` pointer in `interview_sessions` to NULL (ensuring history is preserved even if the user uploads a newer resume).

3. **Performance Optimization (Indexing)**:
   * Multi-column indices `(user_id, date)` on `progress_history` ensure dashboard chart queries load within milliseconds.
   * `JSONB` columns are used for highly structured objects (e.g. `improved_bullets`, `skills_score`), allowing quick traversal using PostgreSQL `jsonb_path_query` features when building recommendation maps.
